# MCP / agent-dispatcher — трудности и решения

Сборник проблем, с которыми реально столкнулись при оркестрации kilo/codex/claude
через `dispatcher.mjs`, и как их решать. Дополняется по мере находок.

---

## 1. `(clean)` / `(no changes)` в отчёте ≠ «агент ничего не сделал» ⭐ главная

**Симптом.** `delegate_kilo` возвращает `status_short: "(clean)"`, `diffstat: "(no changes)"`,
пустой `stdout_tail` — выглядит так, будто агент не выполнил работу.

**Причина.** Агенту в промпте велено делать `git commit`. После коммита **рабочее дерево
чистое**, поэтому `git status --short` пуст (`(clean)`), а `git diff HEAD --stat` пуст
(`(no changes)`). Работа не пропала — она **в коммите**, а не в незакоммиченных правках.

**Решение (применено в dispatcher).** В отчёт добавлены поля:
- `committed` — число новых коммитов за время запуска (HEAD до vs после);
- `commit_log` — `git log --oneline <base>..HEAD`;
- `committed_diffstat` — `git diff <base>..HEAD --stat`.

Теперь видно, что агент закоммитил, даже когда дерево чистое. Реализовано через снимок
`getHead(cwd)` до запуска и `getCommitsSince(cwd, baseSha)` после (обе ветки — успех и ошибка).

**Как ревьюить вручную** (если отчёт старый): не верь только `status_short` —
смотри `git -C <worktree> log --oneline` и `git -C <worktree> diff <base>..HEAD`.

---

## 2. Пустой `stdout_tail` у kilo

**Симптом.** В JSON-отчёте и логе `--- stdout ---` пусто `(empty)`.

**Причина.** `kilocode run` в неинтерактивном режиме печатает мало (или ничего), а результат
оформляет коммитом. Это нормально, не ошибка.

**Решение.** Ориентируйся на `committed`/`commit_log` (см. §1) и на лог-файл `log_path`,
а не на stdout. Для прозрачности можно в промпте просить агента в конце вывести
`git --no-pager diff HEAD~N..HEAD`.

## 3. MCP-сервер не подхватывает правки `dispatcher.mjs` на лету

**Симптом.** Добавил агента/поле в `dispatcher.mjs`, но новый инструмент (`delegate_claude`)
или новые поля не появляются в текущей сессии.

**Причина.** MCP-сервер `agent-dispatcher` (stdio) стартует один раз при инициализации
сессии Claude Code из текущего на тот момент файла. Правки файла = новый код **только для
следующего** запуска сервера.

**Решение.** После изменения `dispatcher.mjs` — **перезапустить сессию Claude Code**
(или MCP-сервер). Проверять можно без сессии:
`echo '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' | node dispatcher.mjs`.

## 4. Самоделегирование Claude (`delegate_claude`) и блокировка прав

**Что нужно.** Отдавать задачи отдельному инстансу Claude как равноправному делегату.

**Реализация.** В `AGENTS` добавлена запись `claude`:
`['-p', prompt, '--permission-mode', 'bypassPermissions']`.
- `-p` — headless-режим (печатает ответ и выходит).
- `bypassPermissions` — **обязателен**: `spawnP` закрывает stdin (`stdio: ['ignore',…]`),
  поэтому любой интерактивный запрос подтверждения **завис бы до таймаута**. Делегату
  нужны те же права, что kilo/codex: править файлы, гонять сборку/тесты, `git commit`.

**Грабли — защитный классификатор Claude Code.** Авто-режим может **заблокировать** запуск
правки, расценив `bypassPermissions` как «автономную петлю агентов без апрув-гейтов».
- Это сознательное решение о доверии: автономный Claude исполняет произвольные команды
  в пределах своего worktree.
- Чтобы разрешить: подтвердить действие вручную или добавить правило в `settings.json`
  (Bash-permission на `node dispatcher.mjs` / на запуск `claude`).

**Рекурсия — уже прикрыта.** В начале `dispatcher.mjs` стоит guard: если выставлен
`AGENT_DISPATCHER_CHILD`, повторный запуск dispatcher отвергается. `spawnP` выставляет
`AGENT_DISPATCHER_CHILD=1` детям, поэтому делегированный Claude **не сможет** поднять ещё
один agent-dispatcher и устроить бесконечную вложенную делегацию.

**Безопасная альтернатива (если не нужен bash у делегата).**
`['-p', prompt, '--permission-mode', 'acceptEdits', '--allowedTools', 'Read,Edit,Write,Grep,Glob']`
— делегат только правит файлы, но тогда он **не сможет сам коммитить** (нет git/bash) —
коммит придётся делать оркестратору. Слабее kilo/codex.

## 5. Расход токенов kilo не логируется

**Симптом.** Нельзя узнать, сколько токенов потратил kilo.

**Причина.** `kilocode` не печатает usage в перехватываемый stdout; dispatcher сохраняет
только exit code, длительность, статус git и хвосты stdout/stderr. (codex печатает
`tokens used N` — это видно в его логе.)

**Решение.** Точные числа kilo — в облачной панели kilocode по сессиям `ses_…` из лога.
Локально доступна только длительность (`duration_s`) как грубый прокси нагрузки.

---

## 6. Нельзя собирать C++ внутри worktree агента (тяжёлый FetchContent)

**Симптом.** Хочется, чтобы агент сам собрал/проверил сборку — но в свежем worktree нет
каталога `build/`, а конфигурация тянет зависимости (GLFW/Box2D/ImGui/GTest/nlohmann)
через FetchContent (одна реконфигурация занимала ~480 с на сетевой выкачке).

**Решение (рабочий процесс).**
- Агенты в промпте: **НЕ запускать cmake**, только править код и коммитить.
- Оркестратор держит **единое** build-дерево в основном чекауте и собирает/тестирует
  там после `cherry-pick` веток агентов. Так зависимости выкачиваются один раз.
- Файлы задач разносим по непересекающимся подсистемам → можно мёржить без конфликтов.

## 7. Параллелизм и блокировки

- `MAX_PARALLEL` (по умолчанию 3) — больше одновременных агентов сервер отклонит
  (`MAX_PARALLEL reached`).
- На каждый `cwd` стоит блокировка (`cwdLocks`): два агента в одном worktree одновременно
  не запустить (`cwd is already locked`). Под каждого параллельного агента — **свой worktree**.

---

## 8. headless-агенты и закрытый stdin

**Общее правило.** `spawnP` запускает агентов с `stdio: ['ignore','pipe','pipe']` — stdin
закрыт. Любой агент, который в неинтерактивном режиме спросит подтверждение/ввод, **зависнет
до таймаута**. Поэтому:
- kilo — `run` (неинтерактивный),
- codex — `exec --skip-git-repo-check`,
- claude — `-p --permission-mode bypassPermissions`.
Все три не должны ждать ввода.

## Памятка по полям JSON-отчёта

| Поле | Что значит |
|------|-----------|
| `status_short` | `git status --short` ПОСЛЕ запуска (пусто = `(clean)`; чисто после коммита агента) |
| `diffstat` | `git diff HEAD --stat` (незакоммиченные правки; пусто после коммита) |
| `committed` | число новых коммитов агента (главный признак выполненной работы) |
| `commit_log` | `git log --oneline base..HEAD` |
| `committed_diffstat` | `git diff base..HEAD --stat` |
| `log_path` | полный лог запуска (`logs/<agent>-<ts>.log`) |
| `timed_out` / `error` | таймаут / текст ошибки |
| `stdout_tail` / `stderr_tail` | хвосты вывода агента |
