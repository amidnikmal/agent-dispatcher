# Операционные уроки оркестрации — 2026-06-15

Реальное поведение `delegate_*` под нагрузкой (сессия с проектом electricity-lab).
Особенно — про «залипания» и решения, которыми их обходили.

## 1. delegate_* — БЛОКИРУЮЩИЙ вызов
Вызов висит весь рантайм агента (до `timeout_sec`, по умолчанию 30 мин, max 2 ч).
Пока он висит — ход оркестратора занят, «мониторить» параллельно нельзя.
**Решение:** ставь КОРОТКИЙ `timeout_sec` (напр. 900 = 15 мин). Тогда залипший агент
ловит SIGTERM→SIGKILL (встроенная эскалация диспетчера) и возвращается, а не висит часами.
Короткий таймаут — это и есть анти-залипание.

## 2. Параллельный запуск: часть вызовов «отклоняется», но агенты всё равно стартуют
Если выстрелить несколько `delegate_*` в одном ходу, харнес часто пропускает 1–2, а
остальные помечает «The user doesn't want to proceed» (гонка на разрешении).
**НО** «отклонённый» вызов мог УЖЕ породить detached-агента, который доработает сам.
Наблюдалось: из 6 запусков «отклонены» 4–5, но реально отработали и закоммитили несколько.
**Решение:** после батча НЕ верь статусам вызовов — проверяй `pgrep` + `git -C <wt> log/status`
по каждому worktree. Работа может идти, хотя вызов «отклонён».

## 3. Агенты НЕНАДЁЖНО коммитят
Наблюдались прогоны 47–64 мин с НУЛЁМ коммитов (или один untracked-файл, без фикса).
**Решение:** оркестратор обязан сам инспектировать каждый worktree, **салвадж**
незакоммиченного (`git diff`/`git stash`), и **сам коммитить + собирать + гонять тесты**.
Никогда не верь «готово» от агента — смотри реальный дифф (см. протокол ревью в CLAUDE.md).

## 4. Агенты НАРУШАЮТ изоляцию worktree
Один агент (cwd = свой worktree) редактировал файлы ОСНОВНОГО checkout напрямую и
оставил `.bak`-мусор. Это ломает основное дерево и сводит на нет смысл worktree.
**Решение:** после КАЖДОГО агента проверяй `git status` основного worktree тоже, не только
его собственного. Чисти `.bak`/мусор.

## 5. НЕ убивай build-процессы агентов небрежно
Агенты порождают `cmake`/`ninja`. `pkill ninja/cmake` наугад оставляет осиротевшие
процессы, которые держат лок ninja, а убийство ПОСРЕДИ реконфигурации повреждает
`FetchContent` `_deps` (исчезает `<dep>-src`) → CMake пытается переклонировать с
заблокированного github → многочасовое зависание. Большая часть «зависаний» сессии —
самонанесённые так.
**Решение:** не убивай сборки наугад; если надо — таргетированно по PID и не во время
configure. Рецепты офлайн-сборки и восстановления `_deps` — в репозитории electricity-lab,
`docs/SESSION_2026-06-15_LESSONS.md`.

## 6. Диагностируй КОРЕНЬ до делегирования
Задача «почини X в модуле A» может на деле жить в модуле B. Пример: «индуктор не связан
с цепью» — это был визуальный баг механической аналогии (маховик без зубьев), а НЕ солвер
(тесты солвера зелёные). Агент с неверным scope сжёг 64 мин, ничего не найдя.
**Решение:** оркестратор сам воспроизводит и локализует корень, и только потом нарезает
задачу агенту с точным scope.

## 7. File-scope дисциплина против конфликтов мержа
Задачи, трогающие ОДИН файл (напр. маховик и пружина — оба в `ProjectionBuilder.cpp`),
НЕЛЬЗЯ давать двум параллельным агентам — гарантированный конфликт. Такие — один агент
последовательно. Параллель — только на непересекающихся наборах файлов.
Для тест-файлов помогает `file(GLOB CONFIGURE_DEPENDS tests/test_*.cpp)`: агенты кладут
новые `test_*.cpp` без правки CMakeLists (убирает конфликт-магнит).

## Итоговый паттерн «жёсткого трекинга»
1. Короткий `timeout_sec` (≈15 мин).
2. Непересекающийся file-scope; один файл = один агент.
3. Засеять `_deps` worktree из здоровой сборки (иначе их `cmake` виснет на сети).
4. Каждый ход: `pgrep` + `git status/log` по каждому worktree → жив / закоммитил / завис.
5. Оркестратор сам коммитит, собирает, гоняет тесты, мержит. Агентам — только черновую работу.
6. Сложные/дизайнерские задачи (где агент уже ломал тесты) — делать самому.
